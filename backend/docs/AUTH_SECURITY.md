# IMUSI Secure Authentication (SQLite)

## 1) Architecture

### Backend modules

- `app/core/config.py`
  - Loads auth/JWT/OAuth settings from environment.
  - Defines access/refresh expiry and auth rate-limit settings.
- `app/core/security.py`
  - Password complexity validation.
  - Argon2/bcrypt hashing and verification.
  - JWT creation/verification for access and refresh tokens.
  - SHA-256 refresh token hashing before persistence.
- `app/models/user.py`
  - `users` table model: local + Google account identity.
- `app/models/refresh_token.py`
  - `refresh_tokens` table model: hashed tokens, expiry, revocation.
- `app/schemas/auth.py`, `app/schemas/user.py`
  - Request/response contracts for auth routes.
- `app/services/auth_service.py`
  - Signup/login, Google code exchange, token issuance, rotation, revocation.
- `app/dependencies/auth.py`
  - `get_current_user` dependency for protected API routes.
- `app/api/routes/auth_routes.py`
  - `/auth/*` endpoints.
- `main.py`
  - Enforces JWT secret presence at startup.
  - Adds auth-specific brute-force rate limit (5/min/IP).

### Mobile modules

- `mobile/src/services/tokenStorage.ts`
  - Secure token storage in Expo SecureStore.
- `mobile/src/services/api.ts`
  - Axios interceptor injects `Authorization: Bearer`.
  - Auto-refresh on `401` with refresh token rotation support.
- `mobile/src/services/authService.ts`
  - Signup/login/logout.
  - Google OAuth code flow via Expo AuthSession.

## 2) SQLite schema (auth)

### `users`
- `id` (PK)
- `email` (unique, indexed)
- `password_hash` (nullable for Google-only users)
- `google_id` (unique, indexed, nullable)
- `name`
- `avatar_url`
- `created_at`
- `updated_at`

### `refresh_tokens`
- `id` (PK)
- `user_id` (FK -> `users.id`)
- `token_hash` (SHA-256 hex, unique, indexed)
- `expires_at` (indexed)
- `revoked` (indexed)
- `created_at`

Additional indexes:
- `(user_id, revoked)`
- `(user_id, expires_at)`

## 3) Authentication flows

### Email/password
1. `POST /auth/signup` with `email`, `password`.
2. Enforce complexity and hash password with Argon2.
3. Create user and issue:
   - access token (15 min)
   - refresh token (30 days default)
4. Persist `sha256(refresh_token)` only.

### Login
1. `POST /auth/login`.
2. Verify hash with Passlib (Argon2/bcrypt).
3. Issue and store token pair as above.

### Refresh rotation
1. `POST /auth/refresh` with refresh token.
2. Verify refresh JWT signature/claims.
3. Lookup stored hash (`sha256(token)`), ensure not revoked/expired.
4. Revoke old token row.
5. Issue new access + refresh token and store new refresh hash.

### Logout
1. `POST /auth/logout`.
2. Hash refresh token and mark row revoked.

### Google OAuth (Expo AuthSession code flow)
1. Mobile opens Google consent screen, receives auth `code`.
2. Mobile sends `code` + `redirect_uri` to `POST /auth/google`.
3. Backend exchanges code with Google token endpoint.
4. Backend fetches Google profile (`sub`, `email`, `name`, `picture`).
5. Create/link user and issue local JWT token pair.

## 4) API contracts

### `POST /auth/signup`
- Request: `{ email, password, name? }`
- Response: `{ access_token, refresh_token, token_type, expires_in, refresh_expires_in, user }`

### `POST /auth/login`
- Request: `{ email, password }`
- Response: same as signup.

### `POST /auth/google`
- Request: `{ code, redirect_uri }`
- Response: same as signup.

### `POST /auth/refresh`
- Request: `{ refresh_token }`
- Response: `{ access_token, refresh_token, token_type, expires_in, refresh_expires_in }`

### `POST /auth/logout`
- Request: `{ refresh_token }`
- Response: `204 No Content`

### `GET /auth/me`
- Header: `Authorization: Bearer <access_token>`
- Response: user profile

## 5) Security controls

- Password attacks:
  - Complexity enforced.
  - Argon2 hashing (bcrypt fallback support).
- Brute force:
  - Auth endpoint rate limit: 5 requests/minute/IP.
- Token theft mitigation:
  - Short-lived access token.
  - Refresh token rotation + revocation.
  - Only hashed refresh token persisted.
- SQL injection:
  - SQLAlchemy ORM / parameterized queries.
- Secrets management:
  - JWT/OAuth credentials only from environment.
  - `JWT_SECRET` and `JWT_REFRESH_SECRET` must each be at least 32 characters.
- Mobile token storage:
  - SecureStore only (no AsyncStorage).

## 6) Setup checklist

1. Install dependencies:
   - Backend: `pip install -r backend/requirements.txt`
   - Mobile: `cd mobile && npm install`
2. Create `backend/.env` from `.env.example` and set secrets.
3. Run migration: `cd backend && alembic upgrade head`
4. Configure Google OAuth app:
   - Add the redirect URI generated by `AuthSession.makeRedirectUri`.
   - Set `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` in backend.
   - Set `GOOGLE_EXPO_CLIENT_ID` in `mobile/src/config/api.ts`.
5. Start backend and mobile app.
